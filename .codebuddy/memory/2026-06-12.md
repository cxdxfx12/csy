## 收费项目添加问题 (续) + SSE 实时推送连接循环修复 (00:13-00:30)

### 问题

  - `[Admin SSE] 实时推送已连接` → `连接断开` 反复循环（每3秒）
  - `V9 Object` 错误重复出现
  - 所有页面添加/编辑的 submitForm API 调用后缺少 catch 导致未处理 reject

### 根因分析
1. **SSE 重连死循环**：`HeaderBar.vue` 的 `onerror` 每次 3 秒固定延迟重连，无退避、无最大次数限制。快速重连导致日志刷屏
2. **submitForm 错误处理缺失**：之前批量修复了 validate 模式，但 API 调用部分缺少 catch。`request.ts` 拦截器在 code !== 0 时 `Promise.reject()`，submitForm 只有 finally 没有 catch → 未处理的 promise rejection → `V9 Object` 错误
3. **SSE 路由缺失**：`route/api.php` 的 `api/admin` 组中缺少 `sse/stream` 路由。config/route.php 设置 `url_route_must=true`，虽然实际可能通过 Nginx 或其他方式工作，但添加路由更可靠
4. 后端 API 测试正常（`curl` 测试 `/api/admin/charge/itemAdd` 返回了正确响应 `{"code":1,"msg":"数据已存在..."}` — 说明后端没问题）

### 修复
1. **SSE 重连退避**（`HeaderBar.vue`）：
   - 添加 `sseRetryCount`、`SSE_MAX_RETRIES=10`、`SSE_BASE_DELAY=2000`、`SSE_MAX_DELAY=60000`
   - `onerror` 改为指数退避：delay = min(2000 × 2^(重试次数-1), 60000)
   - connect 成功时重置 retryCount=0
   - 超过最大重试次数停止连接
   - 移除了冗余的 `error` 事件监听（与 onerror 重复）
   - `disconnectSSE()` 设置 retryCount 阻止重连

2. **ChargeItem submitForm** 添加 catch 块（错误已在拦截器中提示，此处仅防止未处理 reject）

3. **路由补充**：在 `route/api.php` 的 `api/admin` 组中添加 `Route::get('sse/stream', 'admin/AdminSse/stream')`

4. 构建部署 276 个文件全部同步到服务器

## 收费项目添加的真正根因：duplicate code 唯一索引冲突 (00:30-00:38)

### 问题
SSE/前端的修复后用户仍无法添加收费项目。端到端 API 测试发现返回 `{"code":1,"msg":"数据已存在，请勿重复添加"}`（来自 public/index.php 全局异常捕获的 Duplicate entry 翻译）。

### 根因
`ds_charge_item` 表有 `UNIQUE KEY uk_code (code)`，`code` 字段默认值为 `''`（空字符串）。
`ChargeItem::add()` 方法中从不生成 `code` 值，所有新插入的行 `code` 都是 `''`，MySQL 唯一索引冲突 → Duplicate entry 错误。

### 修复
1. **ChargeItem.php add()**：移除 `id` 字段（`unset($data['id'])`），自动生成唯一 code：`CI_` + 时间戳 + 随机后缀
2. **ChargeItem.php edit()**：移除 `id` 字段，保留原 code 值（如被清空则恢复）
3. **数据库**：执行 SQL 修复所有已存在的空 code 行：`UPDATE ds_charge_item SET code = CONCAT('CI_', ...) WHERE code = '' OR code IS NULL`

### 验证
三次连续 API 测试（带 code / 不带 code × 2）全部返回 `{"code":0,"msg":"添加成功"}`

✅ **用户确认修复生效，添加收费项目功能正常** (00:43)
