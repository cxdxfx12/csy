Bug 终结者：系统化调试方法论

告别瞎猜式调试，建立科学的 Bug 定位-修复-预防体系

🔍 Bug 调试的 5 步心法

  1. 精准复现 — 找到稳定的触发条件，记录完整操作序列
  2. 划定范围 — 用二分法/日志插桩缩小可能出错的代码区域
  3. 构建假设 — 基于代码逻辑推断可能的原因，列出2~3个假设
  4. 验证假设 — 逐个验证，不是猜对了就停，要确认没有其他原因
  5. 根治 + 预防 — 修改代码后补测试用例，防止回退

📋 Bug 报告模板：喂给 AI 的最佳格式

模糊描述 = 模糊答案。给 AI 的信息越精确，诊断越快。

标准 Bug 报告模板：

【Bug 标题】一句话描述现象 【环境信息】 - OS: Windows 11 / CentOS 7.9 - PHP: 8.1.12 / Node: 20.11.0 - Framework: ThinkPHP 6.1 / Vue 3.4 - MySQL: 8.0.35 - Browser: Chrome 124 【复现步骤】 1. 登录 admin 账号 2. 进入「业主管理」→ 点击「新增」 3. 填写手机号 138xxxx → 点击保存 4. 观察：页面弹窗报错 【期望行为】保存成功后关闭弹窗，列表刷新 【实际行为】弹窗提示「保存失败」，但未说明原因 【错误日志】 [2026-06-06 21:30:15] ERROR: SQLSTATE[23000]: Duplicate entry '138xxxx' for key 'phone' in /app/admin/controller/Owner.php:82 【相关代码】 [粘贴 Owner.php 第 70~100 行] 【已尝试】 1. 检查数据库确认该手机号已经存在 ✓ 2. 在前端加了手机号唯一校验 ✗ 仍然报错
为什么这个模板有效 AI 一眼就能看到：错误类型（Duplicate entry）、出错位置（Owner.php:82）、已尝试方案（前端校验无效）。诊断几乎瞬间完成。

🎯 常见Bug类型与AI调试策略

1️⃣ 空指针 / Undefined

症状AI诊断Prompt常用修复
Cannot read property 'xxx' of undefined "这段代码中 result.data 可能为 undefined，请分析哪些情况下会触发，并提供防御性代码" 可选链 ?. 、默认值 || []、类型守卫
Call to a member function on null "find() 返回 null 时后续代码会崩溃，请帮我改成先判空再处理" if($obj) {} 提前return

2️⃣ 异步时序问题（前端高频）

症状诊断修复
数据还没返回就渲染了 忘记 await 或 Promise 未处理 加 await / 用 Suspense / 加 loading 状态
先渲染了旧数据才更新 watch 的 immediate/flush 配置不对 检查 watch 选项 / 改用 computed

3️⃣ 数据库相关 Bug

症状诊断方式AI 辅助
查询慢（N+1）开启 SQL 日志，看重复查询"这段代码有N+1查询问题，请用 with() 预加载优化"
死锁SHOW ENGINE INNODB STATUS"多表更新操作，请分析死锁风险并给出加锁顺序"
数据不一致检查是否忘了事务"给这段代码加上事务处理，确保原子性"

4️⃣ 权限 / 安全类 Bug

症状诊断修复
普通用户能访问管理接口中间件/路由守卫遗漏"帮我检查这段路由配置是否有权限漏洞"
用户A能看到用户B的数据查询未加 owner_id 过滤"在这个查询中加入用户归属校验"

🩺 AI 诊断实战案例

案例1：前端路由守卫失效

你
我的 Vue 3 项目路由守卫失效了：即使 Token 过期，用户仍然能访问需要登录的页面。这是我的 router/index.ts 代码：[粘贴]
AI
找到2个问题：
1. 路径匹配不精确：你的 meta.requiresAuth 只标记了部分路由，/admin/users 没有标记
2. Token过期检查缺失：你只检查了Token是否存在，没有检查exp过期时间

修复方案：用正则匹配所有 /admin/* 和 /staff/* 路径，统一校验；在路由守卫中解码JWT检查exp字段。

案例2：生产环境 500 但本地正常

你
上线后「业主缴费」页面返回500，但本地一切正常。Nginx日志显示：FastCGI sent in stderr: "PHP message: PHP Fatal error..." 但没有完整信息。
AI
这是典型的「本地OK/线上炸」问题，排查清单：
1. PHP版本差异：本地8.1，服务器可能是7.4 — 检查nullsafe ?-> 语法
2. Composer依赖版本：生产环境可能缺少某扩展
3. .env 配置：数据库连接/密钥是否正确
4. 文件权限：runtime/ 目录是否可写
5. 中药方子：在入口文件加 ini_set('display_errors',1); error_reporting(E_ALL); 看完整错误
